# v0.70 — Fully-Automated Warmup + Outreach

**Date:** 2026-05-25
**Status:** Planning
**Driver:** Eliminate operator-driven wave clicks. Target: zero human involvement per outreach campaign.

---

## Goal

Run cold-DM outreach campaigns end-to-end without operator clicks. Accounts warm themselves for a week, then run outreach autonomously with captcha solving, proxy diversity, and burn-tolerant operations.

**Volume target:** 5–10 cold DMs per warmed account per day. With 40 accounts surviving warmup, that's 200–400 DMs/day. Per campaign: 1,000–3,000 delivered DMs over 5–10 days.

## Phase 1 — Light active warmup (Days 1–3) — REVISED 2026-05-25

### Why active-from-day-1 (operator decision)
Original plan had 3 days of pure passive IP-soak. Operator pushed back: a dormant account that suddenly turns active looks as suspicious as the IP discontinuity itself. Real users do *something* on Discord every day. Switching to light-active-from-day-1.

### Daily routine per account (Days 1–3)
| Day | Actions |
|---|---|
| Day 1 | Join 1–2 public servers. Post 1 channel message (spintax). |
| Day 2 | Join 1–2 more public servers. Post 1 channel message. Send 1 inter-account DM to cross-proxy partner. |
| Day 3 | Post 1 channel message. Send 1 inter-account DM. Set avatar + bio. Accept any incoming FRs. |

All actions:
- Egress through the account's assigned proxy (v0.66 enforcement)
- Spintax-randomized content
- Timing randomized inside a 1-hour window per day, biased by per-account "active hours" seed
- Captcha-solver fires on any challenged server-join

### Cross-proxy guardrail
Two accounts on the SAME proxy MUST NEVER DM each other. Inter-account DMs are paired ONLY across proxies. Pairings rotate every 2 days. Same-proxy account pair = botnet cluster signal to Discord.

### Code
- Account status `captured → warming` immediately on Day 1.
- `warmup-engine.ts` cron picks accounts every 5 min, rolls the day's action-probability table, executes through proxy, logs to `warmup_actions`.
- 4004 at any point → `quarantined`.

### Risks + mitigations
- **First action through proxy is an IP discontinuity.** Mitigated by choosing low-flag actions for Day 1 (guild joins are common across diverse IPs; cold DM is not).
- **Two accounts on same proxy doing similar actions in same minute.** Schedule with min 5-min stagger between any two actions on the same proxy.
- **Operator manually touches an account from home IP.** Single-account fingerprint reset. We quarantine and retry that one account; others unaffected.

### Exit criteria
- Day 3 end: ≥ 32 of 40 accounts (80%) still in `warming` with no 4004. Transition rule for `warming → warmed` deferred to Day 10 (per Phase 2 below).

## Phase 2 — Full warmup ramp (Days 4–10, 7 days)

### Daily routine per account
Cron fires every 5 minutes; picks a random `warming` account; rolls dice against a probability table of actions:

| Action | Frequency | Captcha-prone? |
|---|---|---|
| Join a public server | 1/account/day, days 4–6 only | High — uses 2Captcha if challenged |
| Post a channel message (spintax content) | 1–2/account/day | Low |
| React to a message in a server | 2–4/account/day | None |
| Set/update avatar | Once at day 4 | None |
| Set/update bio | Once at day 4 | None |
| Set status/presence | 1–2/day | None |
| Accept incoming FRs | Whenever they arrive | None |
| Send DM to a cross-proxy account | 1/account/day on days 6–10 | Low (warm-to-warm) |
| Reply to a DM from a cross-proxy account | Whenever they arrive | None |

All actions:
- Run through the account's assigned proxy
- Spintax-randomized content
- Time-of-day biased (per-account "active hours" seed: e.g. account A is 9–17 EST, account B is 14–22 GMT)
- Min 5-min gap between any two actions on the same account
- Min 5-min gap between any two actions on the same proxy (since 2 accounts share)

### Server list
Operator-curated list of 20–30 large, low-moderation public Discord servers (gaming, crypto, general chat). Stored in `tenant_main.warmup_servers`. Each account joins a random subset of 3–5, biased so any two accounts have ≤ 50% server overlap.

### Cross-proxy DM pairings
For the inter-account DM chatter:
- Each account gets a random "DM partner" from a different proxy
- Pairings rotate every 2 days to avoid the same two accounts always talking
- Message content is spintaxed small-talk: `{hey|yo|sup} {how's it going|what's up|long time}`
- No content that looks like marketing/outreach

### Code
- `warmup-engine.ts` cron — picks accounts, rolls actions, throttles
- `warmup-actions.ts` — implementations: `joinPublicServer`, `postChannelMessage`, `reactToRandomMessage`, `setProfile`, `setStatus`, `sendInterAccountDM`
- `warmup_actions` audit table:
  ```sql
  CREATE TABLE tenant_main.warmup_actions (
    id           bigserial PRIMARY KEY,
    account_id   text NOT NULL REFERENCES tenant_main.discord_accounts(id) ON DELETE CASCADE,
    action_type  text NOT NULL,
    target_id    text,
    success      boolean NOT NULL,
    error        text,
    captcha_solved boolean DEFAULT false,
    occurred_at  timestamptz NOT NULL DEFAULT now()
  );
  ```

### Exit criteria
- Day 10: account has ≥ 7 days of varied activity, ≥ 3 servers joined, ≥ 10 channel messages posted, ≥ 5 reactions, ≥ 2 inter-account DMs. Transition to `warmed`.
- 4004 at any point → `quarantined`.

### Risks + mitigations
- **Same servers across accounts.** Discord can correlate "20 of these new accounts all joined the same 5 servers on the same day." Diversify subset per account, stagger join days (some on day 4, some day 5, some day 6).
- **Cross-proxy DM correlation.** If account A on proxy P1 DMs account B on proxy P2 every day, and account C on P1 DMs account D on P2 every day, that's a pattern. Randomize pairings every 2 days.
- **Captcha rate during server-join.** Expect 30–50% of guild joins to hit captcha. 2Captcha handles ~95% of hCaptcha challenges. Budget: ~$0.10 per account warmup.
- **Phone-verification requirement.** Some servers require phone-verified accounts to post in channels. Skip these. Server list curation should pre-filter.

## Phase 4 — Production outreach (Day 12+)

### Campaign engine modifications
- Refuses any account not in `outreach` state
- Per-account daily cap: 5 DMs (configurable, start conservative)
- Per-account spacing: random `15..30` min between DM-creates
- Per-proxy spacing: min 5 min between any two DM-creates on the same proxy
- Account auto-quarantines on first 4004 in a campaign
- Spintax templates only (no static strings)

### Captcha during outreach
- Every DM-create that returns 429 with captcha challenge → submit to 2Captcha
- Average resolution: 15–30s per challenge
- Budget cap per account per day: $0.50 (50–150 captchas)

### Replenishment alerts
- Daily check: count(`outreach`) accounts. If < 25, fire alert via webhook (Slack/Telegram or in-app banner)
- Operator buys aged marketplace accounts, pastes tokens, system runs them through Phase 0–2

## Risks register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| All 40 existing accounts 4004 during IP-soak | Medium | High | Phase 0 is precisely to surface this early. If > 50% burn, pivot to buying marketplace-aged accounts. |
| Captcha-solver vendor outage | Low | High | Wire both 2Captcha *and* CapMonster as fallback. Or: two 2Captcha accounts as redundancy. |
| Cross-proxy DM correlation flagged | Medium | Medium | Randomize partner pairings, vary message content, vary timing. |
| Public servers ban our accounts en masse | Medium | Medium | Diversify server list, monitor join success rate, swap servers if one is "trapping" us. |
| Discord rolls out a new fingerprinting signal mid-warmup | Low | Critical | Accept as background risk. Run small batches first, scale only after verifying signal-stability. |
| 2Captcha quality dips, DMs land in captcha-loop | Medium | High | Per-account captcha budget cap auto-quarantines runaway accounts. |
| Proxy IPs are already known-bad to Discord | Medium | High | Test each proxy in Phase 0 before assigning. Vendor swap if needed. |
