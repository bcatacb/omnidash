# v0.18 Phase 1 — Playwright per-account contexts for captcha-walled REST sends

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Route the 3 captcha-walled Discord REST endpoints (`POST /channels/:id/messages`, `POST /users/@me/channels`, `PUT /users/@me/relationships/:id`) through a real Chromium browser context per account so Discord sees a genuine browser TLS stack + cookie jar + ServiceWorker chain instead of a server-side `cycletls`-impersonated request.

**Architecture:** Add `playwright-core` + Chromium to the `gg-api` container. New module `discord-browser.ts` manages one `Browser` singleton and one `BrowserContext` per account (own proxy, own UA, own localStorage token). Critical sends call `browserFetch(accountId, url, opts)` which forwards to `page.evaluate(() => fetch(...))` so the request executes inside Chromium. All other endpoints (GETs, /experiments, joinByInvite, etc.) keep using existing `tlsFetch` — they are not captcha-walled. Gateway WebSocket stays on the `ws` package (Phase 2 if needed).

**Tech Stack:** `playwright-core@^1.49` (Chromium only), existing `cycletls`, existing `ws`, Node 20 bookworm-slim base.

The three call sites swap `tlsFetch` for `browserFetch` and pass the account id; nothing else changes about how they parse responses.

## Honest constraints & risks (read before starting)

- **Image size:** Final `gg-api` image goes from ~250 MB to ~700 MB. Build time goes from ~30 s to ~3 min. Once.
- **Memory:** Each warm Chromium context idles around 80–150 MB. With 10 accounts that's ~1 GB. Hetzner VPS has 32 GB; comfortable.
- **Unknown unknown:** If Discord checks that the gateway WS and REST call share a TLS session (cookie binding via the WS handshake), Playwright fixing REST alone won't help. Phase 2 in that case is moving the WS into Playwright via `page.evaluate(() => new WebSocket(...))` — explicitly out of scope here.
- **Test account cool-down:** before measuring whether this fixes captcha, the test account must have a 24–48 h quiet window so existing flags decay. Otherwise we can't tell if v0.18 helped.
- **No mocks for the actual fix:** the value of this change is that Discord sees a real browser. Smoke tests verify the *plumbing* (eval-fetch round-trip, context lifecycle). The actual captcha verdict only comes from live Discord traffic post-deploy.

```typescript
/**
 * Per-account Playwright Chromium contexts for captcha-walled Discord REST.
 *
 * Why this exists: cycletls impersonates Chrome at the TLS layer but Discord's
 * anti-abuse stack also inspects the full browser fingerprint — ServiceWorker
 * presence, navigator fields, cookie jar continuity, paint-timing. We can't
 * fake all of that from Node. Running the captcha-sensitive REST calls inside
 * a real Chromium that has already loaded discord.com and identified via the
 * SPA's normal boot path gives Discord exactly the shape it expects.
 *
 * Scope: ONLY the 3 endpoints Discord captcha-walls go through here:
 *   POST /channels/:id/messages
 *   POST /users/@me/channels
 *   PUT  /users/@me/relationships/:id
 * Everything else (GETs, /experiments, joins, listing) stays on tlsFetch —
 * those are not captcha-walled and don't justify the cost.
 *
 * Lifecycle:
 *   - One Browser singleton (lazy-launched on first browserFetch call).
 *   - One BrowserContext per accountId (lazy-created, kept warm). Each context
 *     has its own proxy URL, localStorage.token, cookie jar.
 *   - One Page per context, navigated to https://discord.com/channels/@me so
 *     Discord's SPA boot has executed (fingerprint, sessions, gateway WS may
 *     also have opened — that's fine, we don't use the in-browser WS for
 *     anything, the Node-side ws is still the source of truth for events).
 *   - Idle contexts (no use for IDLE_CLOSE_MS) close to free memory.
 *   - On SIGTERM, closeAllBrowserContexts() shuts every context + the browser.
 *
 * Disabled with BROWSER_FETCH_ENABLED=0; in that mode browserFetch throws so
 * call sites can decide whether to fall back to tlsFetch or surface the error.
 */
import { chromium, type Browser, type BrowserContext, type Page } from "playwright-core";

const ENABLED = process.env.BROWSER_FETCH_ENABLED !== "0";
const DEFAULT_PROXY_URL = process.env.WEBSHARE_PROXY_URL || "";
const IDLE_CLOSE_MS = 30 * 60 * 1000; // 30 min
const USER_AGENT =
  "Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/124.0.0.0 Safari/537.36";

let browserPromise: Promise<Browser> | null = null;

async function getBrowser(): Promise<Browser> {
  if (!browserPromise) {
    browserPromise = chromium.launch({
      headless: true,
      args: [
        "--no-sandbox",
        "--disable-dev-shm-usage",
        "--disable-blink-features=AutomationControlled",
      ],
    });
  }
  return browserPromise;
}

interface AccountContext {
  ctx: BrowserContext;
  page: Page;
  lastUsedMs: number;
  idleTimer: NodeJS.Timeout | null;
}

const accountContexts = new Map<string, AccountContext>();

function parseProxyUrl(url: string):
  | { server: string; username?: string; password?: string }
  | undefined {
  if (!url) return undefined;
  try {
    const u = new URL(url);
    const server = `${u.protocol}//${u.host}`;
    if (u.username || u.password) {
      return {
        server,
        username: decodeURIComponent(u.username || ""),
        password: decodeURIComponent(u.password || ""),
      };
    }
    return { server };
  } catch {
    return undefined;
  }
}

async function createAccountContext(accountId: string, token: string): Promise<AccountContext> {
  const browser = await getBrowser();
  const ctx = await browser.newContext({
    userAgent: USER_AGENT,
    locale: "en-US",
    timezoneId: "UTC",
    viewport: { width: 1280, height: 800 },
    proxy: parseProxyUrl(DEFAULT_PROXY_URL),
  });

  // Inject the user token BEFORE any discord.com page loads. Discord's SPA
  // reads localStorage.token during boot to skip the login screen.
  await ctx.addInitScript((tok: string) => {
    try {
      (window as any).localStorage.setItem("token", JSON.stringify(tok));
    } catch {
      /* localStorage may not be available on about:blank — Discord page itself works */
    }
  }, token);

  const page = await ctx.newPage();
  // Go to the app shell. We don't actually care about render — only that
  // discord.com cookies, ServiceWorker, and SPA boot have happened so the
  // browser's session looks "real" by the time we eval-fetch.
  await page.goto("https://discord.com/channels/@me", {
    waitUntil: "domcontentloaded",
    timeout: 30_000,
  }).catch((err) => {
    // Don't fail context creation on slow nav — Discord can load up to 20 s
    // on cold residential IPs. As long as the cookie jar is populated by then
    // the next fetch from inside the page will still bind correctly.
    console.warn(`[browser] account=${accountId} initial nav warning: ${err?.message || err}`);
  });

  const entry: AccountContext = { ctx, page, lastUsedMs: Date.now(), idleTimer: null };
  scheduleIdleClose(accountId, entry);
  console.log(`[browser] account=${accountId} context ready`);
  return entry;
}

function scheduleIdleClose(accountId: string, entry: AccountContext): void {
  if (entry.idleTimer) clearTimeout(entry.idleTimer);
  entry.idleTimer = setTimeout(async () => {
    const cur = accountContexts.get(accountId);
    if (!cur) return;
    if (Date.now() - cur.lastUsedMs < IDLE_CLOSE_MS) {
      // Got used during the timer — reschedule.
      scheduleIdleClose(accountId, cur);
      return;
    }
    accountContexts.delete(accountId);
    try { await cur.ctx.close(); } catch {}
    console.log(`[browser] account=${accountId} idle-closed after ${IDLE_CLOSE_MS / 60000} min`);
  }, IDLE_CLOSE_MS);
}

async function getOrCreateContext(accountId: string, token: string): Promise<AccountContext> {
  const existing = accountContexts.get(accountId);
  if (existing) {
    existing.lastUsedMs = Date.now();
    return existing;
  }
  const entry = await createAccountContext(accountId, token);
  accountContexts.set(accountId, entry);
  return entry;
}

export interface BrowserFetchOptions {
  method?: string;
  headers?: Record<string, string>;
  body?: string;
  timeoutMs?: number;
}

export interface BrowserFetchResponse {
  status: number;
  ok: boolean;
  headers: Record<string, string>;
  text(): Promise<string>;
  json(): Promise<any>;
}

/**
 * Run a fetch inside the per-account Chromium page. Caller passes account id
 * + token so we can lazy-init the context. The actual HTTP call is the
 * browser's native fetch, so Discord sees its own cookies, ServiceWorker, and
 * fingerprint headers — exactly what the SPA would send.
 */
export async function browserFetch(
  accountId: string,
  token: string,
  url: string,
  opts: BrowserFetchOptions = {},
): Promise<BrowserFetchResponse> {
  if (!ENABLED) {
    throw new Error("browser-fetch disabled (BROWSER_FETCH_ENABLED=0)");
  }
  const { page } = await getOrCreateContext(accountId, token);

  const evalResult = await page.evaluate(
    async ({ url, method, headers, body, timeoutMs }) => {
      const ac = new AbortController();
      const timer = setTimeout(() => ac.abort(), timeoutMs);
      try {
        const r = await fetch(url, {
          method: method || "GET",
          headers: headers || {},
          body: body ?? undefined,
          credentials: "include",
          signal: ac.signal,
        });
        const text = await r.text();
        const headerObj: Record<string, string> = {};
        r.headers.forEach((v, k) => { headerObj[k.toLowerCase()] = v; });
        return { status: r.status, headers: headerObj, text };
      } finally {
        clearTimeout(timer);
      }
    },
    {
      url,
      method: opts.method,
      headers: opts.headers,
      body: opts.body,
      timeoutMs: opts.timeoutMs ?? 20_000,
    },
  );

  const { status, headers, text } = evalResult;
  return {
    status,
    ok: status >= 200 && status < 300,
    headers,
    text: async () => text,
    json: async () => JSON.parse(text),
  };
}

/** Manually drop an account's context (e.g. on token revocation). */
export async function closeAccountContext(accountId: string): Promise<void> {
  const entry = accountContexts.get(accountId);
  if (!entry) return;
  if (entry.idleTimer) clearTimeout(entry.idleTimer);
  accountContexts.delete(accountId);
  try { await entry.ctx.close(); } catch {}
}

/** Shutdown hook: called from SIGTERM in index.ts. */
export async function closeAllBrowserContexts(): Promise<void> {
  for (const [id, entry] of accountContexts) {
    if (entry.idleTimer) clearTimeout(entry.idleTimer);
    try { await entry.ctx.close(); } catch {}
    console.log(`[browser] account=${id} closed on shutdown`);
  }
  accountContexts.clear();
  if (browserPromise) {
    try {
      const b = await browserPromise;
      await b.close();
    } catch {}
    browserPromise = null;
  }
}

export function browserFetchEnabled(): boolean {
  return ENABLED;
}
```

 * Verifies the eval-fetch round-trip works end-to-end without hitting Discord.

import { browserFetch, closeAllBrowserContexts } from "./discord-browser";

async function main(): Promise<void> {
  const acctId = "smoke-test-account";
  const fakeToken = "smoke-token-not-real";
  const url = "https://httpbin.org/anything";

  const res = await browserFetch(acctId, fakeToken, url, {

    headers: { "content-type": "application/json", "x-smoke": "1" },

/**
 * Real outbound message send + relationships fetch.
 *
 * Sends now route through a real Chromium context per account (discord-browser).
 * Relationships GET stays on tlsFetch — Discord doesn't captcha-wall reads.
 */

import { discordHeaders, tlsFetch } from "./discord-http";
import { browserFetch } from "./discord-browser";

export interface SendResult {
  ok: boolean;
  discordMessageId?: string;
  body?: string;
  sentAt?: string;
  error?: string;
  httpStatus?: number;
}

/** Send a DM to a Discord channel. Returns the freshly-issued Discord message id on success.
 *  Captcha-required responses surface as raw 400 errors — the engine's
 *  classifyDiscordError + panic-pause handles them by stopping the campaign.
 *
 *  Routed through discord-browser per account so Discord sees a real Chromium
 *  fingerprint + cookies + SPA-bound session instead of a Node-side request. */
export async function sendDiscordMessage(
  accountId: string,
  token: string,
  channelId: string,
  body: string,
): Promise<SendResult> {
  if (!accountId || !token || !channelId || !body) {
    return { ok: false, error: "missing accountId/token/channelId/body" };
  }
  const url = `https://discord.com/api/v9/channels/${encodeURIComponent(channelId)}/messages`;
  try {
    const r = await browserFetch(accountId, token, url, {
      method: "POST",
      headers: await discordHeaders(token, true),
      body: JSON.stringify({ content: body, flags: 0 }),
      timeoutMs: 20_000,
    });
    const text = await r.text();
    if (!r.ok) return { ok: false, httpStatus: r.status, error: text.slice(0, 200) || `HTTP ${r.status}` };
    let json: any = null;
    try { json = JSON.parse(text); } catch { /* shouldn't happen for 2xx */ }
    return {
      ok: true,
      discordMessageId: String(json?.id || ""),
      body: String(json?.content || body),
      sentAt: String(json?.timestamp || new Date().toISOString()),
    };
  } catch (err: any) {
    return { ok: false, error: String(err?.message || err) };
  }
}
