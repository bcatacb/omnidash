TokTik C2 — Security & Data-Protection Audit

Node/Express + Playwright + Postgres orchestrator · infrastructure security review
Prepared: 27 Jun 2026
Scope: server API, auth, DB, exposure
Method: read-only code & host audit

Bottom line up front. The C2 API is published to the public internet (http://c2.effortlessmetaphor.org) with no working authentication on its data routes, open CORS, and no TLS, backed by a direct superuser database connection. As configured, anyone who finds the hostname can read and modify every account, lead record (PII), and proxy credential, and trigger message sends — no login required. This should be treated as an active exposure and fixed today.

Findings — Ranked by Severity

1Critical Unauthenticated public API. 68 routes; only /api/auth/me checks a token. There is no auth middleware (app.use is only cors / json / static). Accounts, leads, proxies, conversations and send endpoints are wide open and reachable from the internet via nginx.
2Critical Open CORS. cors() with no allowlist lets any website call the API from a visitor’s browser, enabling cross-site data theft and actions against the running session.
3High Default credentials & broken token. Auth falls back to admin/admin; the issued token is base64("admin:<startup-time>") — not secret, no signature, no expiry.
4High No TLS. nginx serves listen 80 only. Lead PII, proxy logins and the API token traverse the internet in plaintext, exposed to network interception.
5High Direct superuser Postgres. A hand-rolled pg.Pool(DATABASE_URL) connects with full privileges — no row-level security is possible. Any code path or leaked env grants total DB control.
6Medium Identifier SQL injection risk. The query builder parameterizes .eq() values (good) but interpolates column/sort identifiers directly (ORDER BY ${column}, SELECT ${cols}) — injectable if any route accepts user-supplied sort/column names.
7Medium Sensitive data at rest, now exposed. Plaintext proxy credentials and scraped-lead personal data live in the DB that Finding 1 leaves reachable. A breach here is also a third-party PII breach.

What’s Done Right

✓ .env is correctly git-ignored (secrets are not committed).   ✓ .eq() filter values use parameterized queries.   ✓ Clear service/transport separation makes the fixes below localized, not a rewrite.

Remediation

Today (stop the exposure)

  • Add one auth middleware (app.use) that rejects any request without a valid token, mounted before all /api routes; exempt only /api/auth/signin.
  • Lock CORS to your frontend origin only.
  • Put the API behind TLS (nginx 443 + cert) and redirect 80→443; or firewall 4000 so it’s not reachable except via HTTPS.
  • Set strong DEFAULT_USER/DEFAULT_PASS now to kill the admin/admin fallback.

This week (harden)

  • Replace the token with a signed, expiring JWT (server-side secret); stop deriving it from a timestamp.
  • Use a least-privilege DB role for the app instead of the superuser connection string.
  • Allowlist sortable columns against a fixed set before interpolating identifiers; never pass raw input.
  • Encrypt proxy credentials at rest and rotate anything that traversed plaintext HTTP.
Scope note. This review covers security and data-protection only — protecting your infrastructure, credentials, and the personal data you hold. It intentionally does not address platform deliverability or account longevity, which for an automated outreach system depend on the messaging behavior itself, not on configuration changes.
