Telegram Account Ban-Risk Audit

Telegram-Portal mass-outreach platform · infrastructure & deliverability review
Prepared: 26 Jun 2026
Scope: backend code, config, sessions
Method: read-only server audit

Bottom line up front. The sending engine is well-built, but the accounts run with no network isolation — one server IP and one shared API key across the whole fleet — and with no enforced daily limit by default. Combined with cold-messaging scraped strangers, this is why accounts are being limited and banned. Fixing the infrastructure slows the losses; only changing the targeting model stops them.

System Overview

FastAPI + Telethon platform (~14k LOC) running ~100–160 user-account sessions on a single VPS. Core flow: scrape group members → campaign-message those non-contacts, with a staging-group technique to gain DM rights, plus an account “warmup” worker. The send layer is genuinely competent — per-account flood rests, peer-flood strikes that retire burned accounts, a rolling-24h counter, and randomized send jitter. The weaknesses are in account isolation, defaults, and the underlying activity, not the pacing code.

Key Findings — Ranked by Ban Impact

1 Critical Whole fleet shares one IP. Per-account proxies are coded but PySocks isn’t installed, so they can’t load — ~100+ accounts egress from a single datacenter IP. This is the loudest “bot farm” signal Telegram detects.
2 Critical One shared API key for every account. All sessions authorize under a single api_id. A whole fleet on one key reads as coordinated automation and risks a published-key flood ban.
3 High No daily cap by default. The per-account daily limit defaults to 0, and the cap is only enforced when set above 0 — campaigns launched without it message strangers unbounded per day.
4 Medium Fast default cadence. 5-second send interval and a light 15-msg/day mutual warmup are aggressive for cold outreach and short for aging new accounts.
5 High The activity itself. Cold-messaging scraped non-contacts is exactly what Telegram’s spam system targets. Recipient “Report” taps drive the limit→ban pipeline regardless of plumbing.

Recommendations

Isolation & hygiene (slows losses)

  • Install PySocks; assign a unique residential/mobile proxy per account. Biggest single fix.
  • Stop sharing one API key across the fleet; verify the current key isn’t flagged.
  • Enforce a real daily cap — refuse to launch a campaign with limit 0; start low (~20–40/day aged, far less for new).
  • Keep device fingerprints unique & stable per account; avoid re-login churn.

Behavioral (actually durable)

  • Longer, real warmup & aging — weeks of joining/reading groups, profile setup, before any cold send.
  • Slower, human-houred sending with wide randomization; no 24/7 blasting.
  • Vary message content (templating) — identical bodies to many users flag instantly.
  • Shift toward opt-in / warm targeting. The only lever that truly stops the bans.
On the appeal bot: it recovers soft “limited” states well, but it cannot reverse permanent bans (those are email-only and rarely overturned for outreach accounts). With one IP, one shared key, and uncapped cold sends, accounts will keep getting limited — and eventually hard-banned — faster than appeals can recover them. Fix isolation and targeting first; the appeal bot is recovery, not prevention.
